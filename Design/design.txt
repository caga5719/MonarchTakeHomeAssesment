System Design: Invoice GL Categorization Dashboard
Overview
473 Amazon invoice PDFs → parse → AI classify each line item → SQLite → REST API → React dashboard.

Tech Stack
Layer	Choice	Why
Backend	Python + FastAPI	Best PDF parsing libs, pandas for aggregations, async support
PDF Parsing	pdfplumber	Handles structured PDFs well, extracts tables reliably
AI Classification	Claude API (claude-sonnet-4-6)	Line-item GL mapping is a language reasoning task — LLM handles edge cases
Database	SQLite	Zero setup, local, portable, sufficient for 473 invoices
Frontend	React + TypeScript + Vite	Fast dev server, strong typing for API data
Charts	Recharts	Composable, React-native, no D3 overhead
Styling	Tailwind CSS	Quick clean UI
GL/Property Data	openpyxl	Read the provided .xlsx files into DB on startup

Database Schema

-- Loaded from GL List.xlsx on startup
CREATE TABLE gl_accounts (
  code        TEXT PRIMARY KEY,   -- e.g. "6702"
  name        TEXT NOT NULL,      -- e.g. "Repairs & Maintenance"
  category    TEXT,               -- broader grouping if present
  description TEXT
);

-- Loaded from Property List.xlsx on startup
CREATE TABLE properties (
  code    TEXT PRIMARY KEY,       -- e.g. "wrmi"
  name    TEXT,
  address TEXT
);

CREATE TABLE invoices (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  invoice_number TEXT UNIQUE NOT NULL,  -- e.g. "111W-XV4P-3P9H"
  date           DATE,
  property_code  TEXT REFERENCES properties(code),
  po_number      TEXT,
  invoice_gl     TEXT,            -- GL as printed on invoice (invoice-level)
  amount_due     REAL,
  subtotal       REAL,
  tax            REAL,
  processed_at   DATETIME
);

CREATE TABLE line_items (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  invoice_id      INTEGER REFERENCES invoices(id),
  line_number     INTEGER,
  description     TEXT NOT NULL,
  asin            TEXT,
  qty             REAL,
  unit_price      REAL,
  subtotal        REAL,
  tax_rate        REAL,
  -- AI-assigned (may differ from invoice-level GL)
  assigned_gl     TEXT REFERENCES gl_accounts(code),
  gl_confidence   TEXT   -- "high" / "medium" / "low"
);
Data Processing Pipeline

Startup
  ├── Load GL accounts from GL List.xlsx → gl_accounts table
  ├── Load properties from Property List.xlsx → properties table
  └── (Optional) Auto-trigger processing

POST /api/process  (user-triggered or on startup)
  └── For each PDF in /Invoices/ not yet in DB:
        1. pdfplumber → extract invoice header fields
           (invoice_number, date, property_code, gl_code, po#, totals)
        2. pdfplumber → extract line items table
           (description, qty, unit_price, subtotal, tax_rate)
        3. Claude API call (batched per invoice):
           - Input: list of item descriptions + full GL chart
           - Output: { line: N, gl_code: "XXXX", confidence: "high" }
        4. INSERT invoice + line_items into SQLite
        5. Mark as processed
Claude prompt strategy: Send all line items for one invoice in a single call with the GL chart as context. This minimizes API calls (1 call/invoice vs 1 call/line-item) and gives Claude context across items.

API Endpoints

GET  /api/summary
     → { total_invoices, total_spend, property_count, gl_count, items_processed }

GET  /api/gl-breakdown
     → [{ gl_code, gl_name, total_spend, item_count, items: [...] }]

GET  /api/property-breakdown
     → [{ property_code, property_name, total_spend, item_count, items: [...] }]

GET  /api/invoices?page=&search=&property=&gl=
     → paginated invoice list with line items

GET  /api/invoices/:invoice_number
     → single invoice with all line items + assigned GL codes

GET  /api/processing-status
     → { total_pdfs, processed, pending, failed }

POST /api/process
     → triggers background processing job
Frontend Views
Summary Dashboard — KPI cards: total spend, invoices processed, properties, top GL category
GL Spend Breakdown — Bar chart of spend by GL + sortable table with drill-down to items
Items by GL — Grouped accordion table: GL → list of items with invoice links
Items by Property — Grouped accordion table: property code → items + spend
Invoice Explorer — Searchable/filterable table of all invoices; click → view line items with their AI-assigned GL vs invoice-level GL
Processing Status — Progress bar + list of any failed/skipped invoices
Key Design Decisions & Tradeoffs
SQLite over Postgres — Zero config for local demo. Handles 473 invoices + their line items with no performance concern.

One Claude call per invoice (not per line item) — Batching keeps API costs down and lets Claude see all items in context (e.g., if 3 items are clearly maintenance supplies, it can infer more confidently on the ambiguous 4th).

Store invoice-level GL separately from AI-assigned GL — Lets the dashboard highlight discrepancies (where AI disagrees with the invoice GL) as a bonus insight.

Process on demand, not re-process — Check invoice_number uniqueness before processing. Re-running the app doesn't burn API credits on already-processed invoices.

pdfplumber over PyMuPDF — Better at extracting tabular data from PDFs with consistent structure like these Amazon invoices.
